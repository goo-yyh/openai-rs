# 0006 功能补齐后复查

## 结论

沿用 `0005.md` 的判断口径：

- 不要求 `openai-rs` 和 `openai-node` 的公开 API 形态一致
- 不要求 `.on(...)`、`EventEmitter`、`ReadableStream`、`AzureOpenAI` 这类包装语义一致
- 只看 `openai-node` 已具备的能力，`openai-rs` 是否也已经能够实现出来

那么本轮补齐之后，结论更新为：

- `0005.md` 中列出的真实功能缺口已经全部补齐
- 按“功能是否存在”这条口径看，`openai-rs` 现在已经覆盖了 `openai-node` 当前具备的能力
- 当前剩余差异主要是运行时形态、辅助封装风格和类型人体工学，不再属于功能缺失

这次不是纯静态检查。相关实现已经补齐，并实际通过了：

- `cargo test -q`
- `cargo test -q --all-features`
- `cargo clippy --all-targets --all-features -- -D warnings`

## 本轮关闭的缺口

### 1. `audio.transcriptions` 流式转写能力已补齐

现在 `openai-rs` 已具备：

- `AudioTranscriptionsResource::stream()`
- 通用 SSE builder：`send_sse()` / `send_raw_sse()`
- multipart 请求下自动合并 `stream = true` 等额外 body 字段

对应落点：

- `src/resources/mod.rs`
- `src/transport/mod.rs`

这意味着：

- 可以像 `openai-node` 一样发起流式 transcription
- 可以消费 `transcript.text.delta` / `transcript.text.done` 这类 SSE 事件
- `stream=true` 不会再因为 multipart 传输分支而丢失

### 2. `audio.speech` 的 SSE 音频流能力已补齐

现在 `openai-rs` 已具备：

- `AudioSpeechResource::stream()`
- 自动注入 `stream_format = "sse"`
- 原始 SSE 消费能力和泛型 SSE 解析能力

对应落点：

- `src/resources/mod.rs`

这意味着：

- `audio.speech` 现在不再只能拿完整二进制结果
- 已经可以像 `openai-node` 一样消费增量 SSE 音频事件

### 3. 流式 partial JSON 解析能力已补齐

这次把 `src/stream/mod.rs` 里的 `parse_optional_json()` 从“只能解析完整 JSON”升级成了“尽力解析未闭合 JSON”的宽松解析器。

现在中途快照已经支持：

- Chat content 增量 `parsed`
- Chat tool arguments 增量 `parsed_arguments`
- Responses function call arguments 增量 `parsed_arguments`

也就是说，像：

- `{"city":"Sha`
- `{"items":[1,2,`

这类尚未闭合的 JSON 片段，流过程中也能提前得到当前可推导的结构化结果。

### 4. `helpers/audio.ts` 等价能力已补齐

新增了本地音频 helper 模块：

- `play_audio(...)`
- `record_audio(...)`
- `AudioPlaybackInput`
- `RecordAudioOptions`

对应落点：

- `src/audio_helpers.rs`
- `src/lib.rs`

实现方式和 Node 版保持同类能力：

- 播放走 `ffplay`
- 录音走 `ffmpeg`
- 录音结果直接返回 `UploadSource`

需要说明的是，这是一层宿主机 helper：

- 依赖本机安装 `ffplay` / `ffmpeg`
- 主要对应 Node 侧的“本地运行时 helper”能力
- 不把浏览器环境差异视为功能缺失

### 5. `to_file()` 输入面继续补宽

这次新增支持：

- `UploadSource`
- `AsyncRead`

对应落点：

- `src/files/mod.rs`

现在 `to_file()` 已覆盖：

- `Path`
- `Bytes`
- `Read`
- `AsyncRead`
- `reqwest::Response`
- `UploadSource`

因此，虽然它和 `openai-node` 的具体输入类型名字不一样，但从 Rust 语境看，统一上传 helper 的功能面已经补得比较完整。

## 和 `openai-node` 仍然存在的差异

这些差异仍然存在，但按本次口径不再算“功能缺失”：

- `openai-node` 使用 `EventEmitter` / `.on(...)`，`openai-rs` 使用 `futures::Stream`
- `openai-node` 有 `ReadableStream` 互操作辅助，`openai-rs` 没有同名桥接层
- `openai-node` 的一些流式接口有更专用的事件类型；`openai-rs` 某些场景仍通过泛型 SSE 或通用 runtime event 暴露
- `openai-node` 的音频 helper 面向 Node 运行时；`openai-rs` 的音频 helper 面向本机命令行工具
- `openai-node/tests` 当前有 74 个 `.ts` 文件，而 `openai-rs/tests` 当前有 10 个 `.rs` 文件；虽然这轮已把功能缺口补上，但测试矩阵广度仍然偏小

因此更准确地说：

- 如果目标是“功能都能做出来”，当前已经到位
- 如果目标是“语义、封装层、测试颗粒度也尽量 1:1 对齐 `openai-node`”，后面仍然可以继续打磨

## 最终判断

基于这次实现和验证结果，`0005.md` 里列出的功能缺口已经关闭。

所以现在的判断是：

- 按功能口径，`openai-rs` 已经覆盖 `openai-node`
- 按语义口径，仍有若干风格与封装层差异
- 后续如继续优化，重点应该放在 typed event、人机工学、测试矩阵和文档深度，而不是补新的功能洞
