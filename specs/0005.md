# 0005 功能覆盖性复查

## 结论

如果口径改成：

- 不要求 `openai-rs` 和 `openai-node` 的公开 API 形态一致
- 不要求 `.on(...)`、`EventEmitter`、`ReadableStream`、`AzureOpenAI` 这种语义或包装方式对齐
- 只看 `openai-node` 具备的能力，`openai-rs` 是否也已经能做出来

那么当前结论是：

- `openai-rs` 已经覆盖了 `openai-node` 的绝大多数核心功能
- 真正还缺失的东西已经不多了
- 剩余缺口主要集中在少量音频流式能力、流式 partial JSON 解析，以及少数 SDK 附加 helper

也就是说，如果目标只是“把 `openai-node` 能做的事情都做出来”，而不是“把使用方式也做成一样”，那么 `openai-rs` 已经非常接近完整。

本次是静态对照检查，没有重新执行 `cargo test`。

## 本次口径

本次复查只按“功能是否存在”判断，不再把下面这些视为功能缺失：

- `Stream<Item = ...>` 和 `.on(...)` 的差异
- `ClientBuilder + azure_*` 与 `AzureOpenAI` 独立类的差异
- `futures::Stream` 与 `ReadableStream` 桥接形态差异
- helper 名字不同，但最终能力已经能做出来
- 路径参数已经安全编码，只是错误提示不像 Node 那么详细

所以这次文档不会再把这些问题写成“功能还没实现”。

- `skills`
- `videos`
- `beta`

这说明从“资源面”看，已经没有大块 namespace 缺失。

### 2. 主链路 HTTP / SSE / WebSocket 功能已经都有

当前已经具备：

- Chat completions 普通请求

如果只看“能不能调用这条链路”，这些功能大多已经在。

## 真正还缺失的功能

下面这些是按当前口径看，仍然可以算“`openai-node` 有，但 `openai-rs` 还没完全做出来”的功能点。

### P0-1. `audio.transcriptions` 的流式转写能力还没做出来

`openai-node` 的 `audio.transcriptions.create(...)` 支持：

- `stream: true`
- 返回流式 transcription 事件
- 配合 `include=["logprobs"]` 返回流式 logprobs

对应实现和类型主要在：

- `openai-node/src/resources/audio/transcriptions.ts`

而 `openai-rs` 当前在 `openai-rs/src/resources/mod.rs` 中的：

- `AudioTranscriptionsResource::create()`

返回的仍是普通 `JsonRequestBuilder<Value>`，没有单独的流式 builder，也没有 transcription delta/done 事件层。

所以这是一项真实缺失的功能，不是语义差异。

### P0-2. `audio.speech` 的 SSE 音频流能力还没做出来

`openai-node` 的 `audio.speech.create(...)` 支持：

- `stream_format: "sse"`
- 以流的方式返回音频事件

对应定义在：

- `openai-node/src/resources/audio/speech.ts`

而 `openai-rs` 当前的：

- `AudioSpeechResource::create()`

是 `BytesRequestBuilder`，只能拿完整字节结果或完整原始 HTTP 响应，不能做增量 SSE 音频流消费。

因此，`audio.speech` 的“流式语音合成”目前仍未覆盖完整。

### P0-3. 流式 partial JSON 解析能力仍然缺失

`openai-node` 在流式聊天里使用了 partial JSON 解析器：

- `openai-node/src/lib/ChatCompletionStream.ts`
- `openai-node/src/_vendor/partial-json-parser/parser.ts`

它支持在 JSON 还未闭合时，就提前得到：

- 增量 `parsed`
- 增量 `parsed_arguments`

`openai-rs` 当前在 `openai-rs/src/stream/mod.rs` 中使用的是：

- `parse_optional_json()`

这只会在当前文本已经是合法 JSON 时返回解析结果。也就是说：

- 最终完整 JSON 出来后，`openai-rs` 能解析
- 但流进行到中途时，`openai-node` 那种 partial parse 能力还没有

如果目标是“把 `openai-node` 在流式 structured output / tool arguments 上的能力也做出来”，这里还缺一块真实功能。

### P1-1. `helpers/audio.ts` 这类附加音频 helper 还没有

`openai-node` 额外提供了：

- `playAudio(...)`
- `recordAudio(...)`

对应文件：

- `openai-node/src/helpers/audio.ts`

`openai-rs` 当前没有这层辅助能力。

这不是 API endpoint 缺失，但如果把 `openai-node` 附带的 SDK helper 也算进“功能”，这里仍然缺。

### P1-2. `toFile()` 的输入面仍然比 Node 小

`openai-node` 的 `toFile()` 能处理更丰富的输入类型，例如：

- `File`
- `Blob`
- `Buffer`
- `Uint8Array`
- `ReadStream`
- `AsyncIterable`
- `Response`

`openai-rs` 当前已有：

- `Path`
- `Bytes`
- `Read`
- `reqwest::Response`

对应实现：

- `openai-rs/src/files/mod.rs`

所以更准确地说：

- `openai-rs` 已经有统一上传 helper
- 但如果把 Node 的输入矩阵也算作“功能”，还没有完全补齐

这一项优先级不高，但确实仍有余量。

## 有差异，但按本次口径不算功能缺失

下面这些差异在上一版复查里会被列为“未完全覆盖”，但按这次“只看功能”的口径，不再算缺失。

### 1. EventEmitter / `.on(...)` / `.off(...)`

`openai-node` 用 emitter；`openai-rs` 用 `futures::Stream` 和 `final_* / snapshot()`。

这是运行时风格差异，不再算功能缺失。

### 2. `toReadableStream()` / `fromReadableStream()`

这是 Node/browser 互操作包装层差异，不再算核心功能缺失。

### 3. `AzureOpenAI` 独立类

`openai-rs` 已经能通过：

- `azure_endpoint(...)`
- `azure_api_version(...)`
- `azure_deployment(...)`
- `azure_ad_token_provider(...)`

实现同样的 Azure API 能力，因此不再把“没有 `AzureOpenAI` 这个类名”视为功能缺失。

### 4. Zod helper 的名字和形态

`openai-node` 有：

- `zodResponseFormat`
- `zodTextFormat`
- `zodFunction`
- `zodResponsesFunction`

`openai-rs` 没有照搬这套 API，但已经有：

- `json_schema_for<T>()`
- `parse_json_payload<T>()`
- `ToolDefinition::from_schema(...)`
- Chat / Responses `parse::<T>()`

所以如果只看“structured output 和 schema 驱动工具调用能不能做”，当前基础能力已经在。它更像 helper 形态差异，而不是核心功能缺失。

### 5. 路径参数没有 Node 那么强的显式诊断

`openai-rs` 已经做了路径编码，主安全问题已经解决。剩下的是错误提示和诊断强度，不再按功能缺失处理。

### 6. Embeddings 的默认优化行为不同

`openai-node` 会在未指定 `encoding_format` 时默认走 `base64`，然后自动解码成 float32 向量。

`openai-rs` 当前没有这层默认优化和自动解码，但 embeddings endpoint 本身已经可用，用户也可以显式指定返回格式。

因此这更像默认行为差异，不作为当前口径下的核心功能缺失。

## 还需要补的 tests

虽然按功能口径看，剩余缺口已经不多，但测试覆盖仍然明显不够。

当前静态统计：

- `openai-node/tests` 有 `74` 个 `.ts` 文件
- `openai-rs/tests` 有 `10` 个 `.rs` 文件

这意味着：

- `openai-rs` 的主链路已经有一定验证
- 但远远没有达到 `openai-node` 那种资源级和行为级覆盖强度

### P0. 必须优先补的 tests

#### 1. 音频流式能力测试

需要补：

- `audio.transcriptions` 的 `stream=true` contract tests
- transcription delta / done / logprobs tests
- `audio.speech` 的 `stream_format="sse"` contract tests

因为这两块是当前最明确的真实功能缺口。

#### 2. partial JSON 流式解析测试

需要补：

- chat content 未闭合 JSON 的增量解析测试
- tool arguments 未闭合 JSON 的增量解析测试
- responses function call arguments 的增量解析测试

这部分最好参考 `openai-node` 的：

- `tests/lib/ChatCompletionStream.test.ts`
- `tests/_vendor/partial-json-parser/partial-json-parsing.test.ts`

### P1. 强烈建议继续补的 tests

即便不算“功能缺失”，下面这些能力也应该补 tests 固定行为：

- `responses.input_items`
- `responses.input_tokens`
- `realtime.client_secrets`
- `realtime.calls`
- `beta.chatkit.sessions`
- `beta.chatkit.threads`
- `beta.realtime.sessions`
- `beta.realtime.transcription_sessions`
- `uploads.parts`
- `containers.files.content`
- `skills.versions`
- `skills.versions.content`
- `evals.runs`
- `evals.runs.output_items`
- `fine_tuning.jobs.checkpoints`
- `fine_tuning.checkpoints.permissions`
- `webhooks` 错误矩阵
- multipart / form 构造

这些在源码里大多已经有实现，但没有足够的 contract tests 固定行为。

## 当前最准确的判断

按“只看功能，不看语义和 API 形态”的口径，当前结论可以压缩成三句话：

1. `openai-rs` 的核心 API 功能已经非常接近 `openai-node` 完整体。
2. 真正还缺的主要是：
   - `audio.transcriptions` 流式
   - `audio.speech` SSE 流式
   - 流式 partial JSON 解析
   - 少量附加 helper（如 `helpers/audio.ts`、更宽的 `toFile()` 输入面）
3. 最大短板已经不再是资源实现本身，而是 tests 数量和专项行为覆盖还不够。
