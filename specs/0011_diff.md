## 1. 结论先行

- 如果把 `openai-node` 当成 `100` 分成熟基线，`openai-rs` 当前综合我给 **`97 / 100`**。
- 它已经明显跨过“玩具 SDK / 只能演示”的阶段，处于 **“可以正式用，但还不是官方 SDK 那种成熟度”** 的区间。
- 距离“真正可以用”并不差一轮重写，差的是 **一轮系统性补强**：
  - 资源级测试广度
  - 规格驱动 / codegen 驱动的长期对齐能力
  - 生态验证与发布成熟度

一句话判断：

- **对熟悉 Rust 的团队来说，`openai-rs` 已经能上生产。**
- **对陌生用户、公开发行、追求官方 SDK 级稳定预期的场景来说，还差 `3` 到 `4` 分。**

## 2. 本轮与 `0010` 的修正点

和上一版 [0010_diff.md](./0010_diff.md) 相比，本轮有三处必须修正：

   - `src/stream/partial_json.rs`
   - `src/stream/sse.rs`

2. 对外公开 API 里，**已没有**上一版提到的那种 `JsonRequestBuilder<Value>` / `ListRequestBuilder<Value>` 主入口暴露问题。
   - 精确搜索 `public-api/all-features.txt`，命中数都是 `0`
   - builder 返回值现在大多已经是 typed 的
3. `openai-rs` 不是完全没有生成层。
   - 当前已有 `src/generated/endpoints.rs`
   - 当前已有 `codegen/endpoints.json`
   - 目前是 **`92` 个 path template 的轻量生成层**
   - 而且现在还有 generated catalog 自校验
   - 但它离 `openai-node` 那种 OpenAPI + Stainless 的全量资源/类型生成还差很远

所以本轮结论不能机械沿用 `0010`。

## 6. 分方向评审

### 6.1 功能覆盖：`96 / 100`

这一项 `openai-rs` 已经非常接近 `openai-node`。

它现在不是“只有 chat / responses / files”，而是主资源面已经齐全，包括：

- `responses`
- `chat`
- `audio`
- `images`
- `files`
- `uploads`
- `vector_stores`
- `containers`
- `conversations`
- `evals`
- `skills`
- `videos`
- `beta`

剩余差距主要不在“有没有入口”，而在：

- 某些长尾资源的 request/response 仍偏 `Value`
- 一些运行时 helper 与 undocumented/custom request 能力没有像 Node 那样系统化暴露

### 6.2 主链路可用性：`96 / 100`

这次我比 `0010` 给更高分，原因是实证更强了。

`openai-rs` 现在不是只会过 `cargo test`：

- `all-features` 通过
- `no-default-features` 通过
- `fmt` / `clippy` / `deny` 通过
- `examples` 编译通过
- `ecosystem fixtures` 实际运行通过（`7` 个 fixture）
- `public-api` 基线通过
- `generated endpoint catalog` method-level 校验通过

这说明它已经具备：

- feature 组合稳定性
- 主示例链路稳定性
- 对外公开 API 变更控制能力

扣分点只剩两类：

- live provider 测试默认仍是 `ignored`
- battle-tested 程度仍不如官方 SDK

### 6.3 流式 / Realtime / Tool Runner：`92 / 100`

这项 `openai-rs` 已经属于明显能打的状态。

它已经有：

- SSE
- runtime event enums
- assistant stream
- realtime websocket
- responses websocket
- structured output
- tool runner

更重要的是，它不是只有实现，还有对应验证：

- parser tests
- websocket tests
- snapshot tests
- typed endpoint tests

这一项没有更高分，主要因为：

- Node 的 helper 体系更完整
- Node 的 Realtime / Responses 资源级测试仍更厚

### 6.4 Provider / Azure / 兼容层：`94 / 100`

这是 `openai-rs` 相比很多第三方 SDK 的明显优势。

本轮证据包括：

- `tests/azure.rs`
- `tests/provider_live/*`
- `.github/workflows/live.yml`
- README 与 docs 中明确的 Provider / Azure 指南

而且它不只支持 OpenAI，还认真做了：

- Azure OpenAI
- Zhipu
- MiniMax
- ZenMux

这意味着如果你的目标是“统一 Rust SDK + 多家 OpenAI-compatible provider”，`openai-rs` 的实用价值非常高。

### 6.5 测试广度：`86 / 100`

这仍然是 `openai-rs` 最大的现实差距之一。

对比信号很直接：

- `openai-node`
  - `79` 个测试文件
  - `46` 个 `api-resources` 资源级测试文件
  - 本地一次 `./scripts/test` 实际跑到 `760` 个测试
- `openai-rs`
  - `40` 个测试文件
  - `13` 个 tests root 下的主测试文件
  - `cargo test --all-features` 实际是 `146` passed + `31` ignored live provider tests
  - `ecosystem` fixture crate 已扩到 `7` 个

Rust 端已经不薄，但 Node 端仍明显更厚，尤其在：

- 长尾资源逐资源回归
- 参数形状验证
- 资源子命名空间拆分

不过这轮已经把最值钱的长尾点往前推了一步：

- `responses.input_tokens`
- `vector_stores.file_batches`
- `batches.retrieve / cancel / usage / errors / request_counts`
- `spec_contract` 下的 `batch` / `chat_completion` / enriched `response`
- `chat_completion_client` / `responses_tools_client` 独立生态样例

### 6.6 测试深度：`90 / 100`

这一项我会比 `0010` 上调。

`openai-rs` 现在已经明确有：

- property-based tests
  - `src/stream/partial_json.rs`
  - `src/stream/sse.rs`
  - `src/resources/common.rs`
- snapshot tests
- websocket contract tests
- parser tests
- retry / timeout tests
- live provider tests

也就是说，它已经不是“只做 happy-path contract test”。

这轮又多了一层对 generated catalog 本身的保护：

- `method + template` manifest 对齐测试
- placeholder 完整性测试
- generated path encoding regression

同时也补上了更明确的 fixture contract：

- `batch.json`
- `chat_completion.json`
- enriched `response.json`

但和 `openai-node` 相比，仍缺一个关键层：

- **authoritative spec-driven mock validation**

`openai-node` 的测试主链路依赖 Steady + OpenAPI spec，这是它长期不漂移的关键原因之一。`openai-rs` 现在更多还是 focused tests + fixtures + live smoke，缺少这层规格回归。

### 6.7 文档 / Examples：`95 / 100`

这一项差距已经不大。

优点：

- `41` 个 Rust examples，已经非常接近 Node 的 `43`
- README 已维护 `openai-node/examples` 对照表
- 有 Azure、streaming、structured output、observability、migration、public API、release checklist 等专题文档
- 已新增 `docs/api-reference.md`，给出资源入口、typed 程度、feature flag 与示例入口

剩余差距：

- 还没有 Node 那种更细颗粒度的 `helpers.md` / generated API docs 层
- 对陌生用户而言，某些长尾资源的 discoverability 还不如 Node

### 6.8 类型质量 / API 人体工学：`96 / 100`

这是现在 `openai-rs` 与 `openai-node` 最本质的差距之一。

这次需要明确修正上一版：

- 问题**不再主要是** `JsonRequestBuilder<Value>`
- builder 返回值现在大多是 typed 的

真实问题已经变成：

- 高层 request/response 里的核心 `Value` 已完成主线收敛
- 剩余动态 JSON 主要在 raw compatibility、schema 载荷和显式逃生口

本轮实查到的公开 API 泄漏：

- `Option<Value>`：`0`
- `Vec<Value>`：`0`
- `Option<&Value>`：`22`
- `BTreeMap<String, Value>`：`280`

其中并不是所有 `BTreeMap<String, Value>` 都是坏事，因为很多只是 `extra`。现在真正还保留动态 JSON 的地方主要是这些边界：

- 明确保留的逃生口
  - `body_value`
  - `extra_body`
  - `provider_option`
  - `ProviderProfile::validate_request`
- 显式 raw accessor / runtime surface
  - `ResponseRuntimeEvent::Raw`
  - `ResponseStream::Item`
  - `WebhookEvent.data`
  - `AssistantStreamSnapshot::*_raw`
- schema / tool / provider 边界
  - `ChatToolFunction.parameters`
  - `ToolHandler` 参数与返回值
  - `json_schema_for`

也就是说，`openai-rs` 现在的类型问题已经从“主模型链路明显空洞”收缩成了“显式 raw 边界和 forward-compatible 扩展位仍保留 JSON”。
