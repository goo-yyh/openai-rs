# openai-rs Rust 重写实施设计

## 1. 文档目标

本文档用于指导将 `./openai-node` 重写为 Rust 版本 `./openai-rs`。文档覆盖：

- 对标 `openai-node` 的公开接口设计
- Rust 版本的模块设计与目录设计
- 核心功能设计：传输、错误、分页、流式、上传、Webhook、工具调用、Provider 兼容层
- 测试设计与覆盖策略
- 对 `openai-node` 现有测试的充分性评估，以及 Rust 迁移时必须新增的测试
- 智谱、MiniMax、ZenMux 的兼容测试设计

本文档的目标不是直接开始编码，而是先把“做什么、怎么做、如何证明做对了”定义清楚。

## 2. 背景与约束

### 2.1 背景

`openai-node` 当前是一个面向 OpenAI 生态的通用 JavaScript/TypeScript SDK，具备以下特征：

- 公开 API 面较大，资源众多
- 既有同步请求，也有 SSE 流、分页、Multipart 上传、Webhook 校验、Realtime/WebSocket
- 既支持 OpenAI，也支持 Azure 风格的兼容适配
- 代码中存在若干 helper API，例如：
  - `chat.completions.parse`
  - `chat.completions.runTools`
  - `chat.completions.stream`
  - `responses.stream`
  - `toFile`

Rust 版本不应只是“把 HTTP 请求改成 reqwest”；它需要：

- 维持主接口语义与调用体验
- 让类型系统承担更多正确性约束
- 在多 Provider 场景下保持统一接口
- 建立比 `openai-node` 更强的回归测试体系

### 2.2 本次重写的设计原则

- OpenAI 兼容接口优先，Provider 差异通过兼容层隔离
- 公开 API 稳定优先，内部模块可逐步演进
- 流式、上传、错误映射、分页、工具调用作为一等能力设计，不做补丁式追加
- 对第三方兼容 Provider 采用“统一接口 + ProviderProfile + 能力矩阵 + 扩展字段透传”
- 测试优先于实现速度，尤其是流式与兼容层

### 2.3 非目标

- 不要求第一阶段就把全部 Beta/WebSocket/Realtime 功能同时上线
- 不要求 1:1 复制 TypeScript 的命名习惯；Rust API 应保持 idiomatic
- 不要求复刻 TypeScript 类型导出方式；Rust 用模块与类型重导出替代

### 3.2 `openai-node` 现有测试充分性评估

当前仓库测试量并不少。静态盘点结果：

- `.test.ts` 文件约 `70` 个
- `test/it` 总量约 `570`

但“测试数量多”不等于“迁移风险低”。现有测试主要有 3 类：

1. 资源模板测试
2. 核心工具测试
3. 少量快照/流式/兼容测试

#### 3.2.1 现有测试的优点

- `index.test.ts` 与 `lib/azure.test.ts` 已覆盖部分重试、超时、header、auth、baseURL 逻辑
- `streaming.test.ts`、`internal/decoders/line.test.ts` 对 SSE/行解码边界做了较好的基础覆盖
- `webhooks.test.ts` 覆盖了签名、时间窗口、多签名等重要路径
- `uploads.test.ts`、`form.test.ts` 对上传输入类型有一定回归能力
- `lib/parser.test.ts`、`lib/ChatCompletionStream.test.ts`、`lib/ResponseStream.test.ts` 证明 helper/stream 功能并非完全无测试

#### 3.2.2 现有测试的主要不足

1. 大量资源测试是模板化烟囱测试。
   - 典型模式是 `create/retrieve/list/delete` 后做 `asResponse` / `withResponse` 验证。
   - 这类测试只能证明“调用没崩”，不能证明“序列化、路径、header、query、错误语义、provider 差异”正确。

2. 很多资源方法没有深度断言。
   - 常见问题：只验证返回不是 `Response`，没有验证 HTTP method、URL、query、body、Content-Type、Accept、边界值。

3. 二进制/流式边界覆盖不足。
   - `audio/speech.test.ts` 当前是 `test.skip`，等于没有覆盖。
   - 多个流式与 websocket 路径没有足够的异常流、取消流、半包、乱序、服务端 error event 测试。

4. 存在“伪覆盖”。
   - `responsesItems.test.ts` 只有类型兼容意图，运行时几乎是 no-op。
   - 一些测试文件只有 happy path，没有 error path。

5. Provider 兼容层覆盖很弱。
   - 现有主要覆盖 OpenAI 与 Azure。
   - 没有针对智谱、MiniMax、ZenMux 的契约测试与在线验证。

6. 没有为 Rust 迁移最关键的部分建立强保障。
   - 序列化兼容
   - `serde` 反序列化联合类型
   - `reqwest` multipart 行为
   - `tokio` 流取消
   - `WebSocket` 状态机
   - trait/object-safe 工具运行器

#### 3.2.3 结论

`openai-node` 的测试可以作为“回归素材”，但不能直接视为“Rust 重写已充分有保障”。Rust 版本必须补齐以下缺口：

- 更细粒度的请求构造断言
- 更强的流式边界测试
- 更强的错误映射测试
- Provider 兼容层测试
- Snapshot/golden fixture 测试
- 真实 Provider smoke test

## 4. Rust 总体架构设计

### 4.1 设计结论

建议采用“单主 crate + 内部模块分层 + feature gating”的架构，而不是一开始就拆成多个 crate。原因：

- 当前目标是尽快形成稳定 SDK，而不是做微内核
- `openai-node` 公开 API 面较大，过早拆 crate 会增加类型重导出复杂度
- 大部分复杂度在传输、流式、provider 兼容，不在 crate 边界

### 4.3 feature 设计

建议默认 feature 尽量保守：

```toml
[features]
default = ["stream", "multipart", "webhooks", "rustls-tls"]
stream = []
multipart = []
webhooks = []
realtime = ["tokio-tungstenite"]
responses-ws = ["tokio-tungstenite"]
structured-output = ["schemars"]
tool-runner = ["structured-output"]
rustls-tls = ["reqwest/rustls-tls"]
```

理由：

- 让纯 HTTP 用户不用引入 websocket 依赖
- structured output 与 tool runner 可以按需启用
- 对第三方 Provider 的支持不依赖 feature，依赖运行时 ProviderProfile

## 5. 公开 API 设计

### 5.1 Client 与 Builder

Rust 公开入口建议如下：

```rust
use openai_rs::{Client, Provider};

let client = Client::builder()
    .provider(Provider::openai())
    .api_key("sk-...")
    .build()?;

let zhipu = Client::builder()
    .provider(Provider::zhipu())
    .api_key("...")
    .build()?;
```

核心类型：

- `Client`
- `ClientBuilder`
- `Provider`
- `ProviderKind`
- `ClientOptions`
- `RequestOptions`
- `ResponseMeta`
- `ApiResponse<T>`

#### 设计要求

1. `Client` 为轻量 cloneable handle。
2. 内部持有：
   - `reqwest::Client`
   - `Arc<ClientInner>`
   - `ProviderProfile`
3. `ClientBuilder` 支持：
   - `api_key`
   - `api_key_provider`
   - `base_url`
   - `timeout`
   - `max_retries`
   - `default_headers`
   - `default_query`
   - `webhook_secret`
   - `dangerously_allow_browser` 不需要；Rust SDK 默认服务端使用
4. Rust 版保留类似 `.with_options()` 的能力：
   - `client.with_options(|opts| { ... })`
   - 返回新的 `Client`

### 5.2 资源访问风格

建议采用“资源句柄 + 请求 builder”的 Rust 风格：

```rust
let rsp = client
    .chat()
    .completions()
    .create()
    .model("gpt-5.4")
    .messages(messages)
    .send()
    .await?;
```

而不是要求所有字段一次性构造超大 struct。原因：

- `openai-node` 大量请求参数很多
- Rust builder 更适合必填/选填字段混合
- 可通过类型态或构建器校验保证 `model`、`messages` 等必填字段

每个请求 builder 必须提供三类发送方法中的一到两类：

- `send() -> Result<T, Error>`
- `send_with_meta() -> Result<ApiResponse<T>, Error>`
- `send_raw() -> Result<http::Response<Bytes>, Error>`

与 `openai-node` 的 `.asResponse()` / `.withResponse()` 对应。

### 5.4 核心横切类型

#### 5.4.1 错误类型

```rust
pub enum Error {
    InvalidConfig,
    MissingCredentials,
    Api(ApiError),
    Connection(ConnectionError),
    Timeout,
    Stream(StreamError),
    WebSocket(WebSocketError),
    Serialization,
    WebhookVerification,
    ProviderCompatibility,
}
```

并保留状态码子类映射：

- `BadRequest`
- `Authentication`
- `PermissionDenied`
- `NotFound`
- `Conflict`
- `UnprocessableEntity`
- `RateLimit`
- `InternalServer`

#### 5.4.2 分页类型

- `Page<T>`
- `CursorPage<T>`
- `ConversationCursorPage<T>`
- `PageStream<T>`

要求：

- 支持 `has_next_page()`
- 支持 `next_page().await?`
- 支持把分页变成异步迭代

#### 5.4.3 流式类型

- `SseStream<T>`
- `ChatCompletionStream`
- `ResponseStream`
- `RealtimeSocket`
- `ResponsesSocket`

要求：

- 可取消
- 可恢复错误上下文
- 支持增量聚合后的 final snapshot
- 保留 event 级访问与 snapshot 级访问

#### 5.4.4 上传类型

- `UploadSource`
  - `Bytes`
  - `Path`
  - `Reader`
  - `FilePart`
- `MultipartField`
- `FileLike`

Rust 版不应复制 JS 的 `toFile`，而应提供更 Rust 的：

- `UploadSource::from_path(...)`
- `UploadSource::from_bytes(...)`
- `UploadSource::from_reader(...)`

### 5.5 Structured Output 与 Tool Runner

TypeScript 版 `zod` helper 在 Rust 中不应照搬。Rust 版建议：

- `StructuredOutput<T>`
  - `T: serde::de::DeserializeOwned + schemars::JsonSchema`
- `ToolDefinition<TArgs>`
  - `TArgs: DeserializeOwned + JsonSchema`
- `ToolRegistry`
- `ToolHandler`

建议公开接口：

```rust
let rsp = client
    .chat()
    .completions()
    .parse::<MySchema>()
    .model("gpt-4o")
    .messages(messages)
    .send()
    .await?;
```

```rust
let final_rsp = client
    .chat()
    .completions()
    .run_tools()
    .model("gpt-4o")
    .messages(messages)
    .register_tool(weather_tool)
    .send()
    .await?;
```

### 6.3 类型设计约定

每个资源方法对应以下类型族：

- `XxxRequestBuilder`
- `XxxRequest`
- `XxxResponse`
- `XxxListParams`
- `XxxPage`
- `XxxStreamEvent`

对于联合类型较多的资源，如 `responses`、`realtime`，使用：

- `enum` + `#[serde(tag = "type")]`
- 必要时 `#[serde(untagged)]`

关键要求：

1. 所有对外结构体都必须保留未知字段扩展能力。
   - 建议通过 `extra: BTreeMap<String, serde_json::Value>`
   - 仅在确有性能压力时按资源裁剪

2. 对兼容 Provider 的额外字段不能在解析时丢失。

3. 流式 delta 类型与 final object 类型必须分离。

## 7. Provider 兼容层设计

### 7.1 为什么必须做 ProviderProfile

如果只依赖 `base_url` 配置，而没有 ProviderProfile，会出现：

- header 认证差异无法统一
- path/query/body 微调散落在资源层
- 能力差异只能靠文档提醒，无法测试
- 智谱、MiniMax、ZenMux 的兼容字段无法规范承载

因此必须建立 Provider 兼容层。

### 7.2 核心抽象

```rust
pub trait ProviderProfile: Send + Sync {
    fn kind(&self) -> ProviderKind;
    fn default_base_url(&self) -> &'static str;
    fn auth_scheme(&self) -> AuthScheme;
    fn capabilities(&self) -> &'static CapabilitySet;
    fn prepare_request(&self, ctx: &mut RequestContext) -> Result<(), Error>;
    fn adapt_error(&self, rsp: &ErrorResponse) -> Error;
    fn validate_request(&self, endpoint: Endpoint, body: &serde_json::Value) -> Result<(), Error>;
}
```

相关类型：

- `ProviderKind`
  - `OpenAI`
  - `Azure`
  - `Zhipu`
  - `MiniMax`
  - `ZenMux`
  - `Custom`
- `AuthScheme`
  - `Bearer`
  - `ApiKeyHeader`
  - `QueryToken`
  - `WebSocketSubprotocol`
- `CapabilitySet`
- `CompatibilityMode`
  - `Passthrough`
  - `Warn`
  - `Strict`

### 7.3 OpenAI

基线 Provider，无额外适配。

要求：

- `Authorization: Bearer <api_key>`
- 默认 `base_url = https://api.openai.com/v1`
- 完整资源面

### 7.4 Azure

Rust 版需要保留 `openai-node` 的 Azure 语义：

- `api-version` 默认 query
- `endpoint` / `base_url` 二选一
- `deployment` 可注入 `/deployments/{model}`
- 认证支持：
  - `api-key` header
  - AAD token provider

### 7.5 智谱 Zhipu

根据智谱官方 OpenAI 兼容文档可确认：

- 兼容 OpenAI SDK
- 基础 `base_url` 为 `https://open.bigmodel.cn/api/paas/v4/`
- 可用 `chat.completions.create`
- 支持流式
- 支持工具调用
- 支持通过 `extra_body.thinking` 开启 thinking 模式
- 流式增量中会出现 `delta.reasoning_content`

Rust 设计要求：

1. 内置 `Provider::zhipu()`
2. 默认 `base_url` 指向智谱官方兼容地址
3. 保留 `extra_body`，不能因为强类型而丢掉 `thinking`
4. `ChatCompletionChunkDelta` 必须允许解析：
   - `content`
   - `reasoning_content`
   - `tool_calls`
5. `strict` 模式下仅校验“本 SDK 明确知道不兼容的字段”；未知字段不得默认拦截

### 7.6 MiniMax

根据 MiniMax 官方 OpenAI 兼容文档可确认：

- 基础 `base_url = https://api.minimaxi.com/v1`
- 模型如 `MiniMax-M2.7`、`MiniMax-M2.7-highspeed`
- `extra_body.reasoning_split = true` 时，thinking 会放入 `reasoning_details`
- 多轮 tool call 对话中必须回传完整 assistant message
- 原生内容中可能包含 `<think>` 标签，不能丢失
- 部分 OpenAI 参数会被忽略
- `n` 仅支持 `1`
- 不支持图像和音频输入
- 旧版 `function_call` 已废弃，应使用 `tools`

Rust 设计要求：

1. 内置 `Provider::minimax()`
2. 在 `CompatibilityMode::Strict` 下启用 MiniMax 兼容校验：
   - `n != 1` 直接报错
   - image/audio input 报错
   - `function_call` 报警告或报错
3. 在 `Passthrough` 下保留原始字段透传，以避免文档更新导致 SDK 阻塞
4. `ChatCompletionMessage` 与 `ChatCompletionChunkDelta` 必须允许：
   - `reasoning_details`
   - `tool_calls`
   - 原始 `content` 中的 `<think>` 文本
5. tool runner 必须支持“把完整 assistant message 回填历史”模式，不能只回填解析后的 text

### 7.7 ZenMux

根据 ZenMux 官方文档可确认：

- OpenAI 兼容聊天接口：`POST https://zenmux.ai/api/v1/chat/completions`
- OpenAI 兼容 Responses 接口：`POST https://zenmux.ai/api/v1/responses`
- 模型列表接口：`GET https://zenmux.ai/api/v1/models`
- 模型 ID 格式：`<provider>/<model_name>`
- 可直接通过 OpenAI SDK 风格使用 `base_url="https://zenmux.ai/api/v1"`

Rust 设计要求：

1. 内置 `Provider::zenmux()`
2. 默认 `base_url = https://zenmux.ai/api/v1`
3. 模型 ID 不做过度魔改，原样透传
4. 在 `Strict` 模式下校验：
   - model id 需符合 `<provider>/<model_name>` 形式
5. 必须支持：
   - `chat.completions`
   - `responses`
   - `models.list`
   - `stream`
6. 模型路由、供应商路由、fallback 等高级能力不直接硬编码成公开字段，统一放入：
   - `extra_body`
   - 或 `provider_options.zenmux`

### 7.8 Provider 扩展字段设计

为避免兼容接口不断变化，所有请求 builder 都应支持：

- `.extra_header(k, v)`
- `.extra_query(k, v)`
- `.extra_body(k, json)`
- `.provider_option(k, json)`

这样可以：

- 容纳智谱 `thinking`
- 容纳 MiniMax `reasoning_split`
- 容纳 ZenMux 路由或额外控制字段

## 8. 核心功能设计

### 8.1 请求执行链路

统一请求链路：

1. 用户创建 `RequestBuilder`
2. Builder 生成强类型 `RequestSpec`
3. `ProviderProfile::validate_request`
4. `ProviderProfile::prepare_request`
5. 合并默认 headers/query/options
6. 构造 `reqwest::Request`
7. 执行重试/超时/取消控制
8. 按 endpoint 类型解析：
   - JSON object
   - JSON array
   - bytes
   - SSE stream
   - websocket
9. 追加 `ResponseMeta`
10. 返回业务对象或流对象

### 8.2 重试与超时

Rust 版需要保留 `openai-node` 的核心语义：

- 默认 `max_retries = 2`
- 超时可重试
- `429`、部分 `5xx`、网络错误可重试
- 支持 `retry-after`
- 支持 `retry-after-ms`
- 支持取消时立刻停止重试

实现要求：

- `tokio::time::timeout`
- 指数退避 + jitter
- 把最终 retry 次数写入 tracing span
- 可选输出 `x-stainless-retry-count` 对等 header，或改为 Rust 自有 debug header 开关

### 8.3 错误映射

错误映射必须有两层：

1. 通用 HTTP 错误映射
2. Provider 特化错误适配

必须保留：

- 状态码
- message
- provider raw body
- request id
- provider kind

### 8.4 序列化与反序列化

要求：

- 参数 struct 尽量强类型
- 兼容字段保留 `extra_body`
- union response 用 `serde` 显式建模
- 所有 response 类型都支持“宽解析”

其中最容易出错的资源：

- `responses`
- `chat.completions`
- `realtime`
- `webhooks`
- `uploads`

### 8.5 Multipart 上传

Multipart 设计要求：

1. 统一抽象 `UploadSource`
2. 文件名推导规则稳定
3. `bytes`、`path`、`reader`、`mime` 均可指定
4. `None` 字段默认剔除
5. 嵌套对象与数组序列化规则必须有测试锁死

### 8.6 SSE 流

必须单独实现，而不是完全依赖第三方 crate 黑盒行为。原因：

- 需要与 `openai-node` 保持一致的 chunk 边界与增量聚合
- 需要对多字节字符、`\r\n`、空 data 行、`[DONE]` 做稳定处理

核心模块：

- `line_decoder.rs`
- `sse.rs`
- `chat_completion_stream.rs`
- `response_stream.rs`

### 8.7 WebSocket

WebSocket 设计分两类：

- `RealtimeSocket`
- `ResponsesSocket`

要求：

- open/message/error/close 生命周期可迭代访问
- 同时支持回调式与 async stream 式消费
- 保留原始 event
- 具备 send/close/error 包装

### 8.8 Webhook 校验

Rust 版需要完整覆盖：

- header 缺失
- secret 缺失
- timestamp 过旧/过新
- 多签名场景
- HMAC 校验
- payload 解析

要求公开接口：

```rust
client.webhooks().verify_signature(payload, headers, secret, tolerance)?;
let event: WebhookEvent = client.webhooks().unwrap(payload, headers, secret, tolerance)?;
```

### 8.9 日志与可观测性

SDK 内部使用 `tracing`：

- request started
- retry scheduled
- response received
- stream opened
- stream parse error
- websocket state changed

必须默认脱敏：

- `Authorization`
- `api-key`
- provider secrets
- webhook secret

## 11. 验收标准

Rust 版本达到“可用重写”的最低标准：

1. 基础资源与高频资源对标完成
2. ProviderProfile 可切换 OpenAI / Azure / Zhipu / MiniMax / ZenMux
3. 流式能力稳定
4. Multipart 上传稳定

6. 单测、合约测试、快照测试齐备

建议最小验收门槛：

- 单元/合约/快照测试总量不少于 `250`
- 高频资源必须做到每方法至少：
  - 1 个 happy path
  - 1 个 request contract
  - 1 个 error path
- 流式相关至少 `40+` 用例
- Provider 兼容层每个 provider 至少 `10+` 离线测试
- 每个 provider 至少 `3~5` 个在线 smoke test

## 12. 风险与应对

### 风险 1：Responses / Realtime 联合类型过于复杂

应对：

- 优先建立 snapshot fixture
- 把增量类型与最终类型分层
- 对未知字段保留宽解析

### 风险 2：第三方 Provider 兼容接口持续漂移

应对：

- ProviderProfile 采用 `Passthrough` 默认策略
- 在线 smoke test 作为漂移探测器
- 对 provider extra 字段保留透传

### 风险 3：MiniMax / Zhipu thinking 字段形态变化

应对：

- fixture + live 双保险
- 结构体保留 `extra` 字段

### 风险 4：ZenMux 既支持 chat 又支持 responses，模型路由格式特殊

应对：

- 把 model id 校验放进 provider strict mode
- 在线测试同时覆盖 `/models`、`/chat/completions`、`/responses`

## 14. 最终结论

这次 Rust 重写的关键不在“把 JS 改成 Rust”，而在于：

- 用 `ProviderProfile` 把 OpenAI / Azure / Zhipu / MiniMax / ZenMux 收敛成统一接口
- 用 builder + 强类型 response + 流式聚合器稳定地承载大接口面
- 用更强的测试体系替代 `openai-node` 当前偏模板化的资源测试

如果按本文档执行，实现顺序建议是：

1. 先完成 `Client + Transport + ProviderProfile + Error + Stream`
2. 再完成 `chat.completions + responses + uploads + webhooks`
3. 然后补齐 Provider 兼容测试
4. 最后扩展 Beta / Realtime / WebSocket

对你的目标而言，最重要的不是“接口先写全”，而是“高频路径先做对，并且能被测试稳定证明”。
